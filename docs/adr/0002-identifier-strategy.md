# ADR-0002: Identifier strategy — ULIDs for external IDs, encrypted NIN with deterministic hash for joins

- **Status**: Accepted
- **Date**: 14 May 2026
- **Owner**: NSR MIS Architecture Team
- **References**: SAD v0.6 §5.2, §8.3, ADR-0001

---

## Context

The NSR holds personally identifiable data on up to 12 million households and the people inside them. Three identifier problems must be solved together:

1. **External identifiers** that operators, partners, programmes, citizens, and audit logs reference. These appear in URLs, on printed receipts, in SMS messages, in DSA-scoped extracts, and in support tickets.
2. **The Ugandan NIN** (National Identification Number), which is the gold-standard identity issued by NIRA. We must store it (with strong protections) and join on it across modules and across partner systems.
3. **Internal identifiers** for high-volume operational tables (audit events, queue rows, log lines) where the database needs fast indexes and we do not need to expose the values externally.

If we get this wrong, we leak information (sequential IDs reveal volume and ordering), we open ourselves to enumeration attacks, we slow joins on a 12M-row table, or we accidentally expose NIN values in logs.

## Decision

We adopt three classes of identifier with strict rules per class.

### 1. External identifiers: ULID

Every entity that is referenced from outside its owning module uses a **ULID** (Universally Unique Lexicographically Sortable Identifier).

- Format: 26 characters, Crockford base32, e.g. `01HXY7K3B2N9PVQE4M6FZRWS18`.
- Lexicographically sortable by creation time (the first 48 bits encode milliseconds since epoch).
- Generated by the application (Python `ulid-py`), not the database, so the value is known before INSERT.
- No information about volume or ordering is leaked across partitions because the millisecond component plus 80 bits of randomness give effective uniqueness per node.
- ULIDs are case-insensitive on the wire but the canonical form is uppercase. The receipt slip, SMS, and admin console all display the uppercase form.

Apply ULIDs to: `Household.id` (the Registry ID), `Member.id` (the Person ID), `Submission.id`, `ChangeRequest.id` (also used as the citizen-facing reference code), `Grievance.id`, `Referral.id`, `ProgrammeEnrolment.id`, `PMTResult.id`, `MatchPair.id`, `MergeDecision.id`, `StageRecord.provisional_registry_id`, `ConnectorRun.id`, `RawLanding.id`, `ExtractJob.id`, `DataRequest.id`, `DataSharingAgreement.id`, `DataProvisionAgreement.id`, `APIClient.id`, `User.id`, `Role.id`, `AuditEvent.id`.

The provisional Registry ID is assigned at capture (CAPI/Web) or at first mapping (bulk) and transferred to confirmed status on promotion. No re-issue, no churn.

### 2. Identity values: encrypted at rest with a deterministic hash

The Ugandan NIN follows the format `^(CM|CF)[A-Z0-9]{12}$` (14 chars total) and is the strongest cross-system identity signal we have. It must be stored to support NIRA verification, dedup, and partner reconciliation, but it cannot sit in cleartext.

Apply this pattern:

| Column | Type | Purpose |
|---|---|---|
| `member.nin_value` | bytea, AES-256-GCM encrypted with a KMS-managed key | Plaintext NIN, decrypted only by code paths with explicit authorisation |
| `member.nin_hash` | bytea (32), SHA-256 of normalised NIN with a project-wide pepper | Used for equality joins, dedup tier 1, NIRA lookup cache, partner reconciliation |
| `member.nin_status` | enum: `has_card`, `lost`, `not_issued`, `no`, `unknown` | Self-reported status from the questionnaire |
| `member.nin_last4` | char(4), decrypted display value | Masked display in lists and tables ("****1234") |

Rules:

- Reads of `nin_value` write an `AuditEvent` per SAD §8.4. Reads of `nin_hash` do not (the hash is not personal data by itself).
- The pepper is stored in the NITA-U KMS and is identical across all NSR environments holding production-equivalent data. Different from any pepper used by partner systems.
- `nin_hash` is indexed (B-tree). `nin_value` is not indexed.
- The DDUP tier 1 deterministic matcher joins on `nin_hash`, never on `nin_value`.
- NIRA-side identity verification sends the encrypted `nin_value` over mTLS; the response is matched back to the household via the `verification_request_id` ULID, not via the NIN.

The same pattern (encrypted value + hash) applies to: passport number, voter card number, driving licence number, and any other identification document number stored in `IdentificationDocument.doc_number`.

### 3. Internal identifiers: BIGINT for high-volume operational tables

For tables where rows are not referenced externally and the call volume is high, use `BIGINT` auto-incrementing primary keys. These are cheaper to index, faster to join, and do not require ULID generation overhead.

Apply to: queue rows for Celery, internal event-store rows, log rows, internal job state rows. Audit events use ULID (they are referenced externally for compliance), not BIGINT.

## Consequences

### Positive

- **Opaque external identifiers**. ULID prefixes are time-ordered but the random tail prevents volume leakage. URLs cannot be guessed.
- **Stable Registry ID lifetime**. Promotion transfers provisional → confirmed without re-issue. Citizens carrying a receipt slip stay on the same ID forever.
- **Fast joins on identity**. `nin_hash` is a fixed-width 32-byte indexed column. Tier 1 dedup runs at sub-second per record at our target scale.
- **Audit minimisation**. Logs reference `member.id` (ULID) and `nin_hash` (opaque) without ever logging `nin_value`. PII does not leak to log aggregation.
- **Operational cost**. ULID generation in Python is microseconds; the cost is negligible at our throughput.
- **Partner reconciliation** without exchanging cleartext NIN. We can hash both sides with the shared pepper, swap hashes, and compare without sending the NIN over the wire.

### Negative / costs

- **ULIDs are 26 characters**, longer than UUIDs (32 hex without dashes, 36 with) in printed form. On a receipt slip the ULID wraps a line. Mitigation: format as two 13-character groups separated by a space.
- **Encrypted NIN columns** cannot be indexed for substring search. Mitigation: any "find by NIN" path uses the hash, not the encrypted value. Operators cannot search by partial NIN; they must enter the full NIN.
- **Pepper management** is a new operational discipline. Mitigation: pepper lives in KMS with the same rotation policy as any other secret, but rotation also requires a re-hash batch job. Plan one re-hash drill per year.
- **Cross-system NIN matching with partners requires a shared pepper** or a key-derivation protocol. Mitigation: do not share peppers; use a two-party hash protocol (each side hashes locally with their own pepper plus a shared salt per request).

### Risks accepted

- **ULID time-ordering can leak relative timing of records** to a determined attacker who collects many IDs. Mitigation: documented; not a confidentiality concern at our threat model (the data is much more sensitive than the timing).
- **A pepper compromise** would weaken the hash. Mitigation: rotate immediately, batch re-hash all rows, document in the incident playbook.

## Alternatives considered

### A. UUID v4 for external identifiers

Rejected. UUIDs are random with no time component. Index locality is poor at scale (rows from the same day scatter across the B-tree). ULIDs give us most of the same uniqueness with better index behaviour and human-readable time ordering.

### B. UUID v7

Considered. Same time-ordered benefits as ULID. Rejected for now because ULID has wider library support in our Python stack and a more compact display form. Will re-evaluate if a Postgres-native uuid-v7 type lands and proves stable.

### C. Sequential BIGINT for external IDs

Rejected. Reveals volume and ordering. Sequential household IDs let any data recipient infer how many households were registered in a given period in their geography. Information leakage is unacceptable for a registry holding poverty-relevant data.

### D. Encrypt NIN without a hash (use encrypted value for joins)

Rejected. Deterministic encryption with the same key reveals equality of values (which is what we want for joins) but cannot be indexed efficiently as a bytea. Hashing is faster, smaller, and keeps the encrypted value off the hot path.

### E. Tokenise NIN via a vault service

Rejected for Phase 1. Tokenisation services add a network hop and an operational dependency. The encrypted-column-plus-hash pattern gets us the same audit and minimisation benefits at lower operational cost.

## Compliance

- DPPA 2019: data minimisation (only `nin_hash` exposed inside the application; `nin_value` access is audited and authorised).
- SAD §8.3 cryptography requirements (AES-256, KMS-managed keys).
- SAD §5.2 identifier strategy.

## Re-open triggers

1. Postgres-native uuid-v7 type matures and becomes the recommended Django/DRF default.
2. NIRA introduces an identifier format change that breaks the regex or the 14-character assumption.
3. A national tokenisation service is offered by NITA-U.
4. Cross-partner two-party hash protocol becomes a standard requirement under DPPA 2019 implementing regulations.
